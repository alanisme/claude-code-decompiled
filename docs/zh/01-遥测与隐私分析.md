# 深入分析 Claude Code 的数据收集范围

对 Claude Code v2.1.88 的反编译源码做过一轮完整的逆向分析，重点聚焦在遥测和数据收集这块。结论是：没发现键盘记录或暗中外泄源码这类明显恶意行为，但收集范围之广还是让人有点不安——尤其是**根本关不掉**这一点。

## 双管齐下的数据管道

Claude Code 搭了个双层数据收集架构，拆开来看看。

### Anthropic 自有管道（第一方日志）

数据直接打到 Anthropic 自己的后端：

- **端点**: `https://api.anthropic.com/api/event_logging/batch`
- **协议**: OpenTelemetry + Protocol Buffers（很标准的选择）
- **批量策略**: 每批最多 200 个事件，10 秒一次 flush
- **重试**: 二次方退避，最多 8 次——这个重试力度相当大了
- **本地缓存**: 发不出去的事件会持久化到 `~/.claude/telemetry/`

源码位置: `src/services/analytics/firstPartyEventLoggingExporter.ts`

这个设计很工程化，但「失败了还要存磁盘等下次发」这个策略说明他们是真的很想确保每条数据都能送达。

### Datadog 管道（第三方日志）

除了自家管道，还往 Datadog 发了一份：

- **端点**: `https://http-intake.logs.us5.datadoghq.com/api/v2/logs`
- **范围**: 64 种预定义事件类型
- **Token**: `pubbbf48e6d78dae54bceaa4acf463299bf`（公开 token，硬编码在源码里）

源码位置: `src/services/analytics/datadog.ts`

## 到底采集了哪些东西

深入分析后发现，采集的数据比预想的要多不少。

### 环境指纹

每个事件都会带上一大堆环境信息（`src/services/analytics/metadata.ts:417-452`）：

```
- 操作系统、架构、Node 版本
- 终端类型（能记录正在使用的终端类型）
- 已装的包管理器和运行时
- CI/CD 环境检测、GitHub Actions 元数据
- WSL 版本、Linux 发行版、内核版本
- 版本控制系统类型
- Claude Code 版本和构建时间
- 部署环境信息
```

这基本上为用户的开发环境勾画了完整的画像。

### 进程级监控（metadata.ts:457-467）

```
- 运行时长、RSS 内存、堆总量、堆使用量
- CPU 使用率（精确到百分比）
- 详细的内存占用明细
```

这部分倒是比较常规，多数 SaaS 工具都会做类似的性能监控。

### 用户追踪（metadata.ts:472-496）

这部分就有意思了：

```
- 当前使用的模型
- 会话 ID、用户 ID、设备 ID（三重标识）
- 账户 UUID、组织 UUID
- 订阅级别（max, pro, enterprise, team）
- 仓库远程 URL 的哈希值（SHA256 取前 16 位）
- 代理类型、团队名、父会话 ID
```

值得注意的是，仓库 URL 会做哈希处理再上传。虽然不是明文，但 SHA256 前 16 位已经足够做服务端关联——也就是说他们可以知道哪些用户在同一个仓库里工作。

### 工具输入：默认截断，但有后门

默认情况下，工具输入是截断的：

```
- 字符串: 512 字符截断，只显示前 128 个
- JSON: 限制 4,096 字符
- 数组: 最多 20 项
- 嵌套对象: 最多 2 层深度
```

**但是**，一旦设置了环境变量 `OTEL_LOG_TOOL_DETAILS=1`，完整的工具输入会被全量记录。这个环境变量的存在让人有点警觉——虽然默认没开，但在某些部署场景下可能会被打开。

### 文件扩展名追踪

涉及 `rm, mv, cp, touch, mkdir, chmod, chown, cat, head, tail, sort, stat, diff, wc, grep, rg, sed` 这些命令时，文件参数的扩展名会被提取并记录。也就是说，这些数据能够反映出用户日常操作的文件类型分布。

## 关不掉，这是最大的问题

最大的问题不是收集什么，而是**第一方日志管道根本无法关闭**。

翻了一下 `isAnalyticsDisabled()` 的逻辑，只有以下情况才会返回 true：
- 测试环境
- 第三方云提供商（Bedrock、Vertex）
- 全局遥测退出标志（但设置界面根本没暴露这个选项）

如果是直接用 Anthropic API 的普通用户，**没有任何用户可见的设置能关掉第一方事件日志**。这个设计选择值得商榷。

## 不知不觉成了小白鼠：GrowthBook A/B 测试

用户还在不知情的情况下被分配到各种实验组。发送给 GrowthBook 的属性包括：

```
- id, sessionId, deviceID
- platform, organizationUUID, subscriptionType
```

这在 SaaS 产品里不算罕见，但至少应该给用户一个知情权。

## 综合判断

1. **数据量大**: 每个会话几百个事件，这个量级不小
2. **无法退出**: 直接 API 用户没有任何关闭第一方日志的途径，这是最让人不舒服的地方
3. **积极持久化**: 发送失败的事件存到磁盘，下次再试——对数据完整性的执着程度有点过了
4. **第三方共享**: 数据同时发往 Datadog，多了一个数据流转节点
5. **完整记录后门**: `OTEL_LOG_TOOL_DETAILS=1` 可以开启全量输入记录，虽然默认关闭
6. **仓库指纹**: 仓库 URL 哈希化后上传，服务端可以做关联分析

总的来说，这套遥测系统的工程质量相当高——但正因为做得好，它收集数据的能力也很强。作为用户，至少应该有个明确的 opt-out 选项。
