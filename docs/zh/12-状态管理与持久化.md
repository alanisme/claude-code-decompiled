# Claude Code 的状态管理与持久化：它怎么把一个会话长期"养活"

Claude Code 不只是"收一条输入，回一条输出"的 CLI。它要维护会话历史、权限模式、模型选择、MCP 连接、记忆系统、项目状态、UI 展开状态，还要在中断、恢复、压缩、fork session 之后尽量保持一致。

所以它的状态管理天然比普通终端工具重很多。

## 1. 它有两层状态：启动期全局单例 + React 响应式存储

第一层在 `src/bootstrap/state.ts`。这是一个模块级单例，负责在 React 挂载前就可用的状态。

源码里甚至有一句很直白的警告：

```typescript
// DO NOT ADD MORE STATE HERE - BE JUDICIOUS WITH GLOBAL STATE
```

但现实是，这里还是装了很多字段：

```typescript
type State = {
  originalCwd: string
  projectRoot: string
  totalCostUSD: number
  totalAPIDuration: number
  totalAPIDurationWithoutRetries: number
  totalToolDuration: number
  cwd: string
  modelUsage: { [modelName: string]: ModelUsage }
  mainLoopModelOverride: ModelSetting | undefined
  sessionId: SessionId
  parentSessionId: SessionId | undefined
}
```

这个单例层存在的理由很现实：

- 有些状态在 UI 起来之前就得有
- 启动过程本身要写会话文件、配置路径、遥测元数据
- 某些低层模块不应该依赖 React context

初始化阶段还会立刻做 `realpathSync`，把 cwd 规范化。源码注释写得很明白：如果启动时路径没对齐，后面 session 文件可能写在一个目录、查在另一个目录，恢复时直接错位。

## 2. React 层没有用 Redux/Zustand，而是自己写了个极简 store

第二层在 `src/state/store.ts`：

```typescript
export function createStore<T>(
  initialState: T,
  onChange?: OnChange<T>,
): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()
  // ...
}
```

就这么点东西：

- 一个闭包里的 `state`
- 一组 listeners
- `Object.is` 比较避免无效通知
- 一个 `onChange` side-effect bridge

这个选择非常务实。Claude Code 的复杂度已经够高了，如果再叠 Redux middleware、slice、action boilerplate，只会让心智负担更糟。

这里最关键的不是 store 本身，而是 `onChange` 回调。  
真正和外部系统同步的副作用，都从这里出去。

## 3. `onChangeAppState` 是状态副作用的总闸门

`src/state/onChangeAppState.ts` 负责在 AppState 变化时同步外部世界。最典型的是 permission mode 的同步：

```typescript
const prevMode = oldState.toolPermissionContext.mode
const newMode = newState.toolPermissionContext.mode
if (prevMode !== newMode) {
  const prevExternal = toExternalPermissionMode(prevMode)
  const newExternal = toExternalPermissionMode(newMode)
  if (prevExternal !== newExternal) {
    notifySessionMetadataChanged({
      permission_mode: newExternal,
      is_ultraplan_mode: isUltraplan,
    })
  }
  notifyPermissionModeChanged(newMode)
}
```

源码注释提到，之前 permission mode 的变化只在 8+ 条变更路径里的 2 条上被同步，导致 UI 经常不同步。后来才集中收口到这里。

这类注释很说明问题：Claude Code 的状态系统不是一次设计到位的，而是边长边收敛。

## 4. 会话历史不是数组，而是带父指针的链

会话持久化写到：

`~/.claude/projects/<sanitized-cwd>/<sessionId>.jsonl`

每条消息不是简单 append 一个数组元素，而是带：

- `uuid`
- `parentUuid`
- `logicalParentUuid`
- `isSidechain`
- session / cwd / gitBranch / version 等元数据

写入时大致是：

```typescript
const transcriptMessage: TranscriptMessage = {
  parentUuid: isCompactBoundary ? null : effectiveParentUuid,
  logicalParentUuid: isCompactBoundary ? parentUuid : undefined,
  isSidechain,
  // ...
  cwd: getCwd(),
  sessionId,
  version: VERSION,
  gitBranch,
  slug,
}
```

这套设计的核心价值是：**支持分叉**。

如果用户 rewind 对话、重新走另一条路径，老分支不会消失，而是变成 sidechain。  
加载时再从最新的非 sidechain leaf 往回追，重建出当前有效对话链。

这比"存一个扁平 messages 数组"复杂不少，但对真实代理会话是更靠谱的，因为它允许历史分支存在。

## 5. 会话文件不是一启动就建，而是"延迟物化"

`Project` 类里有个 `pendingEntries` 缓冲区。  
在第一条真正的用户/assistant 消息出现前，session JSONL 文件并不会落盘。

这么做是为了避免 ghost session：  
用户打开 Claude Code 看一眼就退出，不应该在磁盘上堆一堆空会话文件。

这是很小但很成熟的设计。很多工具不会处理这种"空启动污染"，久了本地状态目录就变成垃圾场。

## 6. 记忆系统其实分两层：CLAUDE.md 和自动记忆

### CLAUDE.md 体系

由 `src/utils/claudemd.ts` 负责加载，顺序包括：

1. `/etc/claude-code/CLAUDE.md`
2. `~/.claude/CLAUDE.md`
3. 项目里的 `CLAUDE.md` / `.claude/CLAUDE.md` / `.claude/rules/*.md`
4. `CLAUDE.local.md`

支持 `@include`，支持路径匹配 rules，也有 40,000 字符上限。

### Auto-memory / memdir

自动记忆存放在 `~/.claude/projects/<sanitized-cwd>/memory/` 一类目录下，入口文件 `MEMORY.md` 还会被硬限制：

```typescript
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000
```

这个限制非常合理。记忆系统如果没有硬上限，迟早变成上下文窗口污染源。

## 7. Session Memory 是另一条更"自动"的长期记忆线

除了 memdir，Claude Code 还有 `src/services/SessionMemory/sessionMemory.ts` 这套 session memory 机制，会在后台抽取当前会话中的关键信息。

它通常与 auto-compact 配合：

- 当对话达到一定 token 压力
- 且 session memory 已经初始化
- 且满足抽取条件

系统就会在后台更新 session memory，以便之后 compaction 时不必重新总结整段历史。

这其实把"状态持久化"和"上下文管理"直接连起来了。  
Claude Code 明显在往一个方向演化：不是把会话历史全量背着走，而是逐步把重要状态沉淀成更适合长期保留的结构化记忆。

## 8. fork / compact / resume 都要求状态一致性特别强

Claude Code 里有很多不是普通聊天产品会遇到的状态问题：

- `--resume`
- `--continue`
- fork session
- compact boundary
- 清理旧消息后的 GC
- sidechain 排除
- 远端 UI / SDK 同步

比如 compact boundary 写入时，系统不只是记录一条"压缩发生了"，还得保证：

- 旧消息不会在恢复时莫名回流
- tailUuid / preserved segment 信息能让后续加载知道边界
- SDK 和 CLI 对 compaction 的理解一致

这类逻辑一旦不严谨，就会出现最糟糕的 bug：  
用户以为状态已经压缩过，结果恢复时又把旧上下文全部捞回来。
