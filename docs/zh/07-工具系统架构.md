# Claude Code 的工具系统是怎么长出来的

Claude Code 最值得学的部分之一，不是某个单独工具，而是它把 40 多个工具统一纳入同一套注册、校验、权限、执行和结果映射机制。越往里看，越会发现这不是"工具列表"，而是一个很完整的工具运行时。

## 1. 入口先统一：`buildTool`

所有工具最后都要经过 `src/Tool.ts` 里的 `buildTool`：

```typescript
export function buildTool<D extends AnyToolDef>(def: D): BuiltTool<D> {
  return {
    ...TOOL_DEFAULTS,
    userFacingName: () => def.name,
    ...def,
  } as BuiltTool<D>
}
```

这个工厂函数看起来简单，实际上非常关键。它解决了两个问题：

- 所有工具都满足统一契约
- 作者没写的字段会自动落到安全默认值上

默认值本身也很讲究：

```typescript
const TOOL_DEFAULTS = {
  isEnabled: () => true,
  isConcurrencySafe: (_input?: unknown) => false,
  isReadOnly: (_input?: unknown) => false,
  isDestructive: (_input?: unknown) => false,
  checkPermissions: (input, _ctx?) => Promise.resolve({ behavior: 'allow', updatedInput: input }),
  toAutoClassifierInput: (_input?: unknown) => '',
  userFacingName: (_input?: unknown) => '',
}
```

这里最重要的不是 `isEnabled`，而是：

- `isConcurrencySafe` 默认 `false`
- `isReadOnly` 默认 `false`
- `isDestructive` 默认 `false`

也就是说，一个新工具作者如果忘了声明这些属性，系统会把它当成**不能并行、不能视作只读**来处理，而不是乐观放行。这是对的。工具系统里，安全默认值永远比便利默认值重要。

## 2. Tool 接口比看起来大得多

Claude Code 里的工具不是"一个函数 + 一段描述"。完整的 `Tool` 类型包含几十个字段，覆盖：

- 输入 schema
- prompt 描述
- 预校验
- 权限检查
- 并发属性
- 结果映射
- 大结果落盘阈值
- ToolSearch 延迟加载配置
- UI 展示名称
- 自动分类器输入

其中几个最关键：

- `validateInput()`：先拦非法输入，再谈权限
- `checkPermissions()`：工具自身的权限逻辑
- `call()`：真正执行
- `mapToolResultToToolResultBlockParam()`：把 typed result 映射回 API 需要的 `tool_result`
- `prompt()`：告诉模型"这个工具该怎么用"

这套设计的好处很明显：工具不是执行期临时拼出来的对象，而是从定义时就被要求描述自己的行为边界。

## 3. schema 不是装饰品，是系统边界

每个工具都用 Zod 描述输入。更有意思的是，大量 schema 都包在 `lazySchema()` 里，而不是模块加载时立刻构建。

这个选择背后有两层考虑：

- 打断循环依赖
- 把 feature gate / 运行时条件推迟到真正需要时再求值

例如 BashTool 会根据功能开关裁掉某些字段，内部字段 `_simulatedSedEdit` 也不会暴露给模型：

```typescript
const inputSchema = lazySchema(() =>
  isBackgroundTasksDisabled
    ? fullInputSchema().omit({ run_in_background: true, _simulatedSedEdit: true })
    : fullInputSchema().omit({ _simulatedSedEdit: true })
);
```

这里 `_simulatedSedEdit` 特别值得一提。它只允许权限系统内部注入，模型本身不能传。如果放开，模型就有机会伪造一个"看似 harmless 的命令 + 实际写文件"的组合，绕过真实执行路径。

这就是 Claude Code 一贯的风格：不是只做类型校验，而是把 schema 当安全边界的一部分。

## 4. 工具注册表就是 `src/tools.ts`

工具注册没有单独的 JSON manifest，也没有花哨插件 DSL，而是一个很直接的 TypeScript 模块。

核心入口大概是这样：

```typescript
export function getAllBaseTools(): Tools {
  return [
    AgentTool,
    TaskOutputTool,
    BashTool,
    ...(hasEmbeddedSearchTools() ? [] : [GlobTool, GrepTool]),
    ExitPlanModeV2Tool,
    FileReadTool,
    FileEditTool,
    // ...
  ]
}
```

这种写法不算"优雅"，但很实用：

- 一眼就能 grep 到所有内建工具
- feature gate 很好插
- bundler 做 DCE 也更直接

代价是函数会越来越长，而且工具可见性规则会分散成很多条件 spread。

## 5. 真正给模型看的工具，要过一整条过滤链

Claude Code 不是把所有已注册工具直接塞给模型，而是经过多层筛选：

1. `getAllBaseTools()`  
   先做 feature gate 和用户类型过滤

2. `getTools(permissionContext)`  
   去掉 blanket deny 的工具，处理 simple mode、REPL mode、`isEnabled()`

3. `filterToolsByDenyRules()`  
   根据权限规则进一步剔除工具，包括 MCP server 前缀规则

4. `assembleToolPool()`  
   合并内建工具与 MCP 工具，按名字去重，并排序以稳定 prompt cache

最后这步排序很聪明。因为系统提示词和工具列表要吃 prompt cache，工具顺序一旦不稳定，缓存命中率会被白白打掉。源码里明确让 built-in tools 作为稳定前缀，再追加排序后的 MCP tools，避免远端服务器的增减把整块缓存打穿。

## 6. REPL 模式其实是"隐藏原语，改给模型一个超级工具"

REPL mode 是个很有意思的设计。开启后，不是简单"多一个 REPLTool"，而是把模型可见的原始工具隐藏掉：

- Bash
- Read
- Edit
- Glob
- Grep
- Write

然后只暴露一个 `REPLTool`。

也就是说，Claude Code 在某些模式下会主动收窄模型的工具表面，把原先分散的能力折叠进一个更强的执行容器里。这样做的好处是：

- 减少模型在工具选择上的负担
- 降低 prompt 体积
- 在内部场景里给工程师更强的一体化执行环境

这其实已经不是单纯的"加工具"，而是在做工具抽象层的重构。

## 7. 权限模型不是外挂，而是工具调用链的一部分

当一个 tool use 到来时，执行链大致是：

1. Zod schema parse
2. `validateInput()`
3. pre-tool-use hooks
4. 权限规则求值
5. `tool.checkPermissions()`
6. classifier / sandbox 相关检查
7. 必要时用户确认
8. `tool.call()`
9. 结果映射与落盘

这个顺序很合理。最关键的是：**先 validate，再 prompt 用户**。  
否则模型传一个明显非法的输入，用户还得先批准一次，这种交互会非常蠢。

对 BashTool 而言，这条链更长。它会继续做：

- 命令解析
- compound command 拆分
- 只读分类
- allow/deny 规则匹配
- classifier 判断
- sandbox 决策

也就是说，权限系统不是一个"统一拦截器"，而是统一规则 + 工具特化逻辑共同作用。

## 8. `runToolUse()` 把执行过程做成了完整流水线

`src/services/tools/toolExecution.ts` 的 `runToolUse()`，几乎就是工具运行时的中央调度器。

大致流程：

1. 通过名字解析工具
2. 检查是否已 abort
3. schema 校验
4. 工具级校验
5. shallow clone 输入并做 observable backfill
6. 执行 hooks
7. 做权限决策
8. 真正执行 `tool.call()`
9. 把结果转成 `tool_result`
10. 大结果必要时写盘，返回预览

这里有个很容易被忽略的细节：backfill 是在输入的浅拷贝上做的，不污染原始输入。  
这么做主要是为了 prompt cache 稳定。用户或模型送来的原始输入尽量保持字节级一致，而给 SDK/UI 看的是展开后的可观察版本。

这类"为了缓存命中率做结构隔离"的处理，在 Claude Code 里反复出现。

## 9. 两个最能代表复杂度的工具

### BashTool

这是最复杂的工具，没有之一。因为 shell 语义太开放，系统只能靠很多层保守约束去"驯服"它：

- tree-sitter 解析命令 AST
- compound command 分段校验
- 识别 read/search/list 型命令
- 后台任务逻辑
- `sleep` 模式特判
- `sed -i` 拦截成模拟编辑路径
- sandbox 决策

最值得学的是 `sed -i` 那套。用户批准的不是抽象"运行一条 sed 命令"，而是一个可预览的具体 diff，随后系统通过 `applySedEdit()` 直接把批准过的结果写盘。这样"批准的内容"和"实际执行的内容"才一致。

### FileEditTool

FileEditTool 走的是相反路线：能力很窄，但可预测性极强。

它不鼓励正则魔法，也不鼓励按行号盲改，而是坚持 **exact string replacement**。  
核心约束包括：

- 先读后改
- `old_string !== new_string`
- 文件不能自上次读取后被修改
- 匹配必须唯一，除非 `replace_all`
- 引号风格会尽量保留

这套策略在"模型不总是精确"的前提下非常有价值。给它一把窄但稳定的手术刀，往往比给一把万能但不可控的瑞士军刀更安全。
