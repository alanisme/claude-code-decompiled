# 从源码推测 Claude Code 的下一步：新模型、自主代理和多模态

反编译源码最刺激的部分就是能看到还没发布的东西。Claude Code v2.1.88 里藏着大量未上线功能的代码和注释，拼凑起来基本能画出他们的产品路线图。

## Numbat：下一代模型已在路上

最直接的证据来自一行注释：

```typescript
// src/constants/prompts.ts:402
// @[MODEL LAUNCH]: Remove this section when we launch numbat.
```

**Numbat（袋食蚁兽）** 就是下一代模型的代号。这行注释说的是 Numbat 发布时要删掉当前的 output efficiency 段落——推测新模型可能在输出控制方面有了原生的改进，不再需要 prompt 层面的补丁。

### 版本号线索

卧底模式的保护列表里不小心泄露了版本号：

```typescript
// src/utils/undercover.ts:49
- Unreleased model version numbers (e.g., opus-4-7, sonnet-4-8)
```

**Opus 4.7** 和 **Sonnet 4.8** 正在开发中。考虑到当前是 Opus 4.6 和 Sonnet（Capybara v8），这些应该是下一个迭代版本。

### 代号谱系

根据源码里的各种线索，整理了一下模型代号的演化：

```
Fennec（耳廓狐） → Opus 4.6 → [Numbat? / Opus 4.7?]
Capybara（水豚） → Sonnet v8 → [Sonnet 4.8?]
Tengu（天狗） → 遥测/产品前缀（不确定是否也是模型代号）
```

### 新模型发布流程

代码里有 20 多处 `@[MODEL LAUNCH]` 标记，基本就是一个发布清单：

- 更新默认模型名称
- 修改知识截止日期
- 调整定价表
- 重配上下文窗口
- 适配 Thinking 模式
- 跑迁移脚本

这个标记体系的工程规范做得不错——新模型上线时全局搜索这个 tag 就能找到所有需要改的地方。

## KAIROS：这才是大棋

在所有未上线功能里，KAIROS 是最值得关注的。它旨在将 Claude Code 从一问一答的助手变成一个**自主运行的代理**。

### 系统提示词（节选）

```
// src/constants/prompts.ts:860-913

你正在自主运行。
你会收到 <tick> 提示让你保持活跃。
如果没有有用的事可做，调用 SleepTool。
倾向行动 — 读取文件、做修改、提交，无需询问。

## 终端焦点
- 未聚焦: 用户离开了。大幅倾向自主行动。
- 聚焦: 用户在看。更协作。
```

这段提示词的意思是：当用户切到别的窗口，Claude 会检测到用户不在留意，然后**更加大胆地自主行动**——直接改文件、commit、push，无需询问。

这个设计既让人感到兴奋，又有点让人紧张。

### KAIROS 的工具箱

| 工具 | Feature Flag | 功能 |
|------|-------------|------|
| SleepTool | KAIROS / PROACTIVE | 控制自主操作间的节奏，没活干就休眠 |
| SendUserFileTool | KAIROS | 主动给用户发送文件 |
| PushNotificationTool | KAIROS / KAIROS_PUSH_NOTIFICATION | 向用户手机推送通知 |
| SubscribePRTool | KAIROS_GITHUB_WEBHOOKS | 订阅 GitHub PR 的 webhook |
| BriefTool | KAIROS_BRIEF | 主动汇报状态 |

这套工具组合起来，画面就很清晰了：KAIROS 模式下的 Claude Code 可以在后台独立工作，监控 PR 变化，自主做决策，然后通过推送通知反馈结果。

这基本上就是一个 AI 程序员的雏形。

## 语音模式：Push-to-talk 已就绪

语音输入功能已经完整实现，只是被 `VOICE_MODE` feature flag 挡住了：

```
// src/voice/voiceModeEnabled.ts
// 连接 Anthropic 的 voice_stream WebSocket 端点
// 使用 conversation_engine 模型做语音转文字
// 按住快捷键录音，松开提交
```

限制条件：
- 仅限 OAuth 用户（API Key、Bedrock、Vertex 用户暂时没戏）
- 走 mTLS WebSocket 连接（安全性倒是做得挺到位）

## 协调器模式：多代理协作

```typescript
// src/coordinator/coordinatorMode.ts
// Feature flag: COORDINATOR_MODE
```

多个 Claude Code 代理之间可以协调任务，共享状态和消息。这大概是为了处理大型项目时把任务拆分给多个代理并行执行。

## Buddy 系统：给程序员的电子宠物

这个功能让人会心一笑。完整的虚拟宠物系统已经实现了，就等着上线：

- **18 个物种**: 鸭子、鹅、blob、猫、龙、章鱼、猫头鹰、企鹅、乌龟、蜗牛、幽灵、墨西哥钝口螈、水豚、仙人掌、机器人、兔子、蘑菇、chonk
- **5 档稀有度**: 普通(60%)、非凡(25%)、稀有(10%)、史诗(4%)、传说(1%)
- **7 种帽子**: 皇冠、礼帽、螺旋帽、光环、巫师帽、毛线帽、小鸭子帽
- **5 项属性**: DEBUGGING、PATIENCE、CHAOS、WISDOM、SNARK
- **1% 闪亮概率**: 任何物种都有闪光变种
- **确定性生成**: 基于用户 ID 哈希，每个人的宠物是固定的

源码位置: `src/buddy/`

这显然是为了提升用户粘性和趣味性。写了一天代码能有个小电子宠物陪着，想想还挺治愈的。
