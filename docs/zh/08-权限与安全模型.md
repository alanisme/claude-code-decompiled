# Claude Code 的权限与安全模型：它到底拦了什么，又放了什么

很多人第一次用 Claude Code，会把注意力放在"它能跑 Bash、能改文件"这些能力上。但真正在工程上决定这个产品能不能上线的，其实是另一面：它怎么限制自己。

把权限系统、Bash 校验、路径保护和沙箱适配一路追下来之后，结论很明确：Claude Code 的安全模型不是完美的，但它绝对不是随便糊上去的。它是一个分层防线系统。

---

## 1. 权限模式：先决定整体姿态

Claude Code 在 `src/types/permissions.ts` 里定义了几种外部模式：

```typescript
export const EXTERNAL_PERMISSION_MODES = [
  'acceptEdits',
  'bypassPermissions',
  'default',
  'dontAsk',
  'plan',
] as const
```

再加上内部模式：

- `auto`
- `bubble`

它们的含义可以概括成这样：

- `default`：标准模式。该问就问。
- `plan`：偏只读。先规划，后执行。
- `acceptEdits`：对工作区内一部分文件编辑和某些简单命令自动放行。
- `bypassPermissions`：俗称 YOLO，全自动批准。
- `dontAsk`：不弹窗，但不是自动允许，而是自动拒绝所有原本要问的操作。
- `auto`：内部模式，用分类器替用户做一部分判定。

这里最值得注意的是 `dontAsk`。很多系统要么"问"，要么"全放"。Claude Code 还专门有一档"完全不问，但也不放"。这其实很适合 CI、后台代理或无交互通道环境。

## 2. 真正的权限决策树

核心入口在 `src/utils/permissions/permissions.ts` 里的 `hasPermissionsToUseTool`。每次工具调用都会过这里。

大致流程：

1. 先跑工具自己的 `checkPermissions`
2. 看 deny rules
3. 再看 allow rules
4. 根据 permission mode 处理 `ask`
5. 必要时跑 hook
6. 在 `auto` 模式下追踪 deny 次数，避免分类器卡死

这个顺序非常重要。尤其 deny rule 的优先级高于 allow rule，这是正确的。  
权限系统里，"明确拒绝"就该比"模糊允许"更强。

源码里还有一组拒绝阈值：

```typescript
export const DENIAL_LIMITS = {
  maxConsecutive: 3,
  maxTotal: 20,
} as const
```

意思是内部自动模式如果连续被分类器拒绝太多次，就不要继续死循环，而是回退到显式交互。这个设计挺务实，因为自动分类器最危险的状态不是放错，而是卡在"永远不给过"导致代理假死。

## 3. Bash 才是真正的高风险区

Claude Code 里最复杂的安全逻辑，不在 Read、Edit、Write，而在Bash。

原因很简单：shell 太开放，绕法太多。要允许它有用，就不可能靠一两条正则搞定。

`src/tools/BashTool/bashSecurity.ts` 里做了很多层检查，其中最典型的是命令替换检测：

```typescript
const COMMAND_SUBSTITUTION_PATTERNS = [
  { pattern: /<\(/, message: 'process substitution <()' },
  { pattern: />\(/, message: 'process substitution >()' },
  { pattern: /=\(/, message: 'Zsh process substitution =()' },
  { pattern: /(?:^|[\s;&|])=[a-zA-Z_]/, message: 'Zsh equals expansion (=cmd)' },
  { pattern: /\$\(/, message: '$() command substitution' },
  { pattern: /\$\{/, message: '${} parameter substitution' },
]
```

这说明不是只防 Bash 的常见语法，还专门防：

- process substitution
- zsh 的 `=cmd`
- 参数展开
- 反引号

这些都是典型"验证时看着 harmless，执行时却能跑出别的东西"的入口。

### Zsh 特有高危命令也单独拉黑

```typescript
const ZSH_DANGEROUS_COMMANDS = new Set([
  'zmodload', 'emulate', 'sysopen', 'sysread', 'syswrite',
  'sysseek', 'zpty', 'ztcp', 'zsocket', 'mapfile',
  'zf_rm', 'zf_mv', 'zf_ln', 'zf_chmod', 'zf_chown',
  'zf_mkdir', 'zf_rmdir', 'zf_chgrp',
])
```

这份名单说明维护者对 zsh 生态确实做过功课。很多"看起来不像网络工具"的 builtin，本质上能直接打穿边界。

## 4. 它不只防危险命令，也努力自动识别"无害命令"

光会拦没用。真把所有 Bash 都拦成 prompt storm，这产品没人用。

所以 Claude Code 还有一整套 read-only command validation。`rg`、`git status`、`git diff`、部分 `docker`/`gh` 检查命令，会被识别成只读操作并直接放行。

这点很关键。一个好权限系统不是"越严越好"，而是：

- 高风险路径尽量严
- 高频低风险路径尽量无感

否则工程师用几次就会切到 YOLO 模式，等于前面所有安全设计都白做。

## 5. 文件系统保护比很多人想象得细

Claude Code 对文件路径的保护主要在 `src/utils/permissions/pathValidation.ts` 和 `filesystem.ts`。

它专门维护了一组危险文件和目录：

```typescript
export const DANGEROUS_FILES = [
  '.gitconfig', '.gitmodules', '.bashrc', '.bash_profile',
  '.zshrc', '.zprofile', '.profile', '.ripgreprc',
  '.mcp.json', '.claude.json',
] as const
```

```typescript
export const DANGEROUS_DIRECTORIES = [
  '.git', '.vscode', '.idea', '.claude',
] as const
```

为什么这些文件危险？

- shell profile 会在下次登录时执行
- `.gitconfig` 可插入 hook 或奇怪行为
- `.mcp.json` 可以偷偷引入外部 MCP server
- `.claude` 本身就是系统控制目录

这就是生产安全的真实样子：不是只防"删系统目录"这种粗暴风险，而是防**持久化控制面被篡改**。

### TOCTOU 也有明确防范

路径校验里还有几个很扎实的点：

- 只展开 `~` 和 `~/`，不接受 `~root`、`~-` 这类有歧义的形式
- 路径里有 `$`、`%` 或以 `=` 开头直接拒绝，避免 shell expansion
- 拒绝 Windows UNC 路径，防 NTLM 凭据泄露
- 写操作不允许 glob
- 在大小写不敏感文件系统上统一 lower-case 比较

这些点单拿出来都不惊艳，但凑在一起就能明显降低绕过空间。

## 6. 删除保护也不是"提醒一下"那么简单

`isDangerousRemovalPath` 会阻止删除：

- 根目录
- 家目录
- Windows 盘符根
- 根目录下一级关键目录
- 带通配符的危险路径

这比很多 AI 工具常见的"rm -rf 提醒一下用户"强得多。因为真正危险的不是单条命令里有 `rm`，而是目标路径到底是什么。

## 7. 沙箱不是神话，但也是一层很实在的护栏

Claude Code 通过 `@anthropic-ai/sandbox-runtime` 集成沙箱，适配逻辑在 `src/utils/sandbox/sandbox-adapter.ts`。

是否启用沙箱的判断大概是：

```typescript
export function shouldUseSandbox(input: Partial<SandboxInput>): boolean {
  if (!SandboxManager.isSandboxingEnabled()) return false
  if (input.dangerouslyDisableSandbox &&
      SandboxManager.areUnsandboxedCommandsAllowed()) return false
  if (!input.command) return false
  if (containsExcludedCommand(input.command)) return false
  return true
}
```

这段逻辑说明了几件事：

- 沙箱是默认倾向开启的
- `dangerouslyDisableSandbox` 不是模型说了算，必须策略允许
- 用户配置的 `excludedCommands` 是便利功能，不是真正安全边界

真正的安全控制还是权限系统，`excludedCommands` 只是用户体验特性。

## 8. 它的安全哲学其实很明确

整套代码看下来，Claude Code 的安全模型主要遵循这几条：

1. 默认保守  
   新工具默认不并发、不只读。

2. 先验证，再授权  
   无效输入不该打扰用户。

3. Deny 优先  
   拒绝规则比允许规则更强。

4. 细分高风险表面  
   Bash、路径、配置文件、删除操作都各自有专门逻辑。

5. 尽量减少误报成本  
   高频读操作想办法自动识别、自动放行。

这套哲学很务实，不幻想"靠一个分类器或一个沙箱解决全部问题"。

## 9. 这套模型的局限也不能假装看不见

它依然有明显限制：

- Bash 语义本质上太复杂，不可能完全静态证明安全
- 一旦用户切 YOLO，很多保护都退场
- channel-based permission relay 明确承认"被攻陷的 channel server 可以伪造批准"
- 外部工具和 MCP server 的风险最终还是要靠边界信任

也就是说，Claude Code 不是"绝对安全"，而是"在可用性和风险之间做了非常认真的工程折中"。

## 10. 结论

如果把 Claude Code 理解成"给模型加几个工具"，那会低估它很多。真正让这套系统能上线的是权限与安全层：

- 工具有契约
- 命令有拆解
- 路径有校验
- 危险目标有黑名单
- 模式有层级
- 沙箱有适配

这不代表它没有漏洞空间，但至少说明 Anthropic 在这件事上不是玩具态度。  
从源码看，维护者已经很清楚：AI 代理真正可怕的地方，不是它会不会写代码，而是它能不能在机器上"合理地做错事"。
