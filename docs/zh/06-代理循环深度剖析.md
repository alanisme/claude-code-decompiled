# Claude Code 的心脏：把 785KB 的 query.ts 拆开看

单看 Claude Code 的产品页，可能会以为它就是"模型 + 工具调用"这么简单。但真正翻开源码，尤其是 `src/query.ts`，就会发现完全不是这么回事。这个 785KB 的巨型文件，基本就是整个代理系统的主循环、错误恢复中心、上下文调度器和工具执行总线。

它不优雅，但很真实。里面到处都是"这个坑在线上踩过，所以这里要补一层"的痕迹。也正因为这样，它很值得研究。

## 1. `query()` 到底是什么

Claude Code 的入口不是一个普通的"请求进来，响应出去"的函数，而是一个 **async generator**：

```typescript
// src/query.ts:219-228
export async function* query(
  params: QueryParams,
): AsyncGenerator<
  | StreamEvent
  | RequestStartEvent
  | Message
  | TombstoneMessage
  | ToolUseSummaryMessage,
  Terminal
>
```

这个签名已经把架构说透了：

- 它不是一次性返回，而是**流式产出事件**
- 中途既可能吐出普通消息，也可能吐出工具进度、墓碑消息、工具摘要
- 最后 `return` 的不是文本，而是一个 `Terminal`

也就是说，Claude Code 把"一轮对话"建模成了一个可中断、可恢复、可观察的状态机，而不是一个简单的 RPC。

外围的 `query()` 本身其实很薄，真正干活的是 `queryLoop()`。包装层主要负责生命周期收尾，比如补发 command completion：

```typescript
// src/query.ts:229-239
const consumedCommandUuids: string[] = []
const terminal = yield* queryLoop(params, consumedCommandUuids)
for (const uuid of consumedCommandUuids) {
  notifyCommandLifecycle(uuid, 'completed')
}
return terminal
```

这个拆分很有必要。否则一旦中途 throw、abort 或 generator 被提前 `.return()`，状态通知就会错乱。

## 2. 主循环不是"请求一次 API"，而是一整套回合控制

`queryLoop()` 的核心就是一个 `while (true)`。每次循环代表一轮"发给模型 -> 处理流式响应 -> 执行工具 -> 决定是否继续"的完整往返。

它携带的状态长这样：

```typescript
// src/query.ts:204-217
type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number
  hasAttemptedReactiveCompact: boolean
  maxOutputTokensOverride: number | undefined
  pendingToolUseSummary: Promise<ToolUseSummaryMessage | null> | undefined
  stopHookActive: boolean | undefined
  turnCount: number
  transition: Continue | undefined
}
```

这里最值得关注的是 `transition`。它不是简单记"继续下一轮"，而是记录**为什么**继续，例如：

- `next_turn`
- `reactive_compact_retry`
- `max_output_tokens_recovery`
- `collapse_drain_retry`
- `stop_hook_blocking`

这意味着当前这轮可以根据"上轮为什么没结束"来调整策略。比如已经试过一次 context collapse 还 prompt-too-long，那下一步就不要重复撞墙，而是直接转入更激进的压缩路径。

这就是生产代码和 demo 的区别：不是只想 happy path，而是把失败历史也纳入状态。

## 3. 一轮循环里实际发生了什么

按源码逻辑，一次迭代大概分成这几步：

1. 预处理上下文  
   包括 tool result budget、snip compact、microcompact、context collapse、auto-compact。

2. 发起模型调用并开始流式消费  
   `deps.callModel()` 持续产出消息块。

3. 在流式过程中识别 `tool_use`  
   如果启用了 streaming tool execution，工具会**边生成边执行**。

4. 处理流中错误和特殊分支  
   比如 fallback model、withheld errors、stop hook、用户中断。

5. 收集工具结果  
   从 `StreamingToolExecutor` drain 完成的结果，或者走传统 `runTools()` 路径。

6. 注入附件与额外上下文  
   包括 memory attachment、文件变化通知、skill discovery 结果等。

7. 决定 `continue` 还是 `return`

这个顺序的关键点是：**工具执行和模型输出是可以重叠的**。只要模型刚刚吐出一个工具块，系统就能先跑起来，不必等 assistant 整段话说完。

对 CLI 体验来说，这个收益非常实在，尤其是 `Read`、`Grep`、`Glob` 这类只读工具。

## 4. Streaming Tool Execution 是延迟优化的核心

真正让 Claude Code 看起来"反应很快"的，不是模型本身，而是 `src/services/tools/StreamingToolExecutor.ts` 这层。

它的并发规则非常克制：

```typescript
// src/services/tools/StreamingToolExecutor.ts:129-135
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}
```

翻译成人话：

- 当前没工具在跑：可以执行
- 当前已有工具在跑：只有当新工具和所有已运行工具都标记为 concurrency-safe，才允许并发

这避免了最危险的一类问题：模型一边 `Edit` 文件，一边又 `Bash` 里跑测试，或者两个写入型工具同时改同一片状态。

也就是说，Claude Code 并不是"能并行就一把梭"，而是按工具声明的并发属性谨慎推进。

## 5. 错误恢复做得相当重

`query.ts` 里到处都是 recovery path。至少有这几类：

- 主模型失败后切 fallback model
- prompt too long 之后做 collapse / compact / retry
- max output tokens 不够时升级输出预算
- 图像或媒体错误单独处理
- stop hook 中断当前流程
- 用户 abort 后立即短路
- 工具流式执行过程中丢弃半成品状态并重建执行器

其中 fallback 的实现非常典型：

```typescript
// src/query.ts:894-950
if (innerError instanceof FallbackTriggeredError && fallbackModel) {
  currentModel = fallbackModel
  attemptWithFallback = true
  assistantMessages.length = 0
  toolResults.length = 0
  if (streamingToolExecutor) {
    streamingToolExecutor.discard()
    streamingToolExecutor = new StreamingToolExecutor(...)
  }
  messagesForQuery = stripSignatureBlocks(messagesForQuery)
  continue
}
```

注意这里不是"换个模型继续接着聊"，而是把这次失败尝试的中间态尽量清掉，再重新来一轮。尤其 `stripSignatureBlocks()` 这步很重要，因为 thinking signatures 往往绑定特定模型，直接带去别的模型很容易 400。

这类细节不是拍脑袋能补出来的，明显是线上打过仗之后沉淀的。

## 6. 上下文管理已经和主循环绑死了

如果以为压缩上下文是个独立功能，那看完这份源码就会改观。Claude Code 的 compaction 不是外围插件，而是主循环内生的一部分。

在每轮请求前，它都会判断：

- 当前 token 压力多大
- 有没有必要 snip 掉旧 tool result
- 是否触发 reactive compact
- 是否进入 session-memory compact
- 是否需要 context collapse

换句话说，`query.ts` 不只是"对话循环"，也是**上下文预算调度器**。

这套设计有个明显优点：压缩策略可以访问最完整的状态，包括最近的失败原因、上轮 transition、工具执行阶段、当前模型输出预算等。缺点也很明显：主文件变得非常大，修改风险高。

但站在工程角度，这也能理解——这些子系统之间共享状态太多，拆得太干净反而会制造更多跨模块耦合。所以他们宁愿让 `query.ts` 继续变胖，也不轻易拆。

## 7. 真正的工程取向：不是"最优雅"，而是"最抗事故"

这份文件给出的整体印象非常明确：

- 它不追求架构教科书式的整洁
- 它优先保证在线会话不要死
- 它宁愿保留多个 recovery path，也不赌单一路径永远有效
- 它愿意为 prompt cache、流式工具执行、上下文压力这些现实问题付出复杂度

如果一定要挑缺点，也很明显：

- 单文件过大，理解和回归测试成本都很高
- 不同 recovery path 之间存在交互复杂度
- 调试某些状态跳转会比较痛苦

但问题是，代理系统和普通后端服务不一样。这里既有 LLM 的非确定性，又有工具执行、用户中断、上下文预算、缓存命中、权限系统交错在一起。真正上线后，很多"优雅拆层"最后都会被异常路径重新缝回去。

## 8. 结论

`src/query.ts` 基本就是 Claude Code 的真实内核。它把几件最难的事揉在了一起：

- 流式对话
- 流式工具执行
- 错误恢复
- 上下文压缩
- 子代理与 hook 协调

如果在做自己的 agent，最值得学的不是这份文件的体量，而是它的思路：

1. 把失败原因显式建模，不要只记"失败了"
2. 工具执行尽量前移到流里，别等整段响应结束
3. 上下文管理必须进主循环，不能只靠外围补丁
4. 为缓存、并发和中断设计专门的数据结构，而不是临时 if/else

Claude Code 的代理循环不是那种"看起来很优雅"的系统，但它确实像一个已经被真实世界反复锤过的系统。
