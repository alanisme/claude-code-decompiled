# Claude Code 如何和上下文窗口死磕

用过 Claude Code 做稍微复杂一点的任务，很快就会体会到一个现实：上下文窗口不是"够不够大"的问题，而是"再大也会被用满"。

一个大文件读下来就是几千到上万 token；多做几轮 `grep + read + edit + verify`，上下文条就会一路冲到 80% 以上。Claude Code 的代码里明显能看到，Anthropic 在这件事上已经不是"加个 compact 按钮"那么简单，而是围绕上下文预算搞出了一整套工程体系。

## 1. 真正的问题不只是用户说了多少话

Claude Code 的上下文消耗，主要来自几部分：

- 系统提示词本身
- 工具定义
- CLAUDE.md / MEMORY.md
- 用户消息
- assistant 回复
- `tool_use` 块
- `tool_result` 块
- 扩展 thinking
- 附件和文件元数据

尤其工具型对话特别吃窗口。一次"读文件 -> 改文件 -> 跑测试"最少会产生：

- assistant `tool_use`
- user `tool_result`
- assistant `tool_use`
- user `tool_result`
- assistant `tool_use`
- user `tool_result`

还没来得及讨论架构，token 已经被操作轨迹吃掉一大截。

默认上下文窗口定义在：

```typescript
export const MODEL_CONTEXT_WINDOW_DEFAULT = 200_000
```

某些模型支持 1M context，会通过 `[1m]` 后缀判断。但就算是 1M，也不是无限，更不是免费的。

## 2. token 计数不是每次都去问 API

Claude Code 用的是"最后一个精确值 + 后续增量估算"的策略。核心函数：

```typescript
export function tokenCountWithEstimation(messages: readonly Message[]): number {
  // ...
}
```

思路很实用：

- 先从最近一次 API 响应里的 usage 取到较准确 token 数
- 再对这之后新增的消息做 rough estimation

粗估算法也很朴素：

```typescript
export function roughTokenCountEstimation(
  content: string,
  bytesPerToken: number = 4,
): number {
  return Math.round(content.length / bytesPerToken)
}
```

普通文本和代码按 4 char/token 估，JSON 更紧按 2。  
图片和 PDF 则直接给一个保守常数。

这做法的优点非常明确：

- 不需要每轮都额外打 tokenizer API
- 对 hot path 足够快
- 对自动压缩这种阈值判断已经够用

这类系统里，精确到个位 token 往往没意义，关键是别为了"更准一点"把整条链路拖慢。

## 3. 他们连"并行工具调用导致的消息切片"都考虑到了

`tokenCountWithEstimation()` 里有一段很能说明问题：  
当一次 assistant 响应里触发多个工具调用时，流式代码会把内容拆成多个 `AssistantMessage`，它们共享同一个 `message.id`，中间还会穿插 tool result。

如果计数逻辑只看最后一个 assistant message，就会漏掉前面同一响应切出来的兄弟块以及中间插进来的结果。

所以源码会往回走，找到同一个 response id 的最早兄弟记录，再从那里开始估。  
这就是典型的"真的做过并行工具流，才会知道这里能踩坑"的代码。

## 4. 自动压缩的阈值不是拍脑袋定的

自动压缩核心在 `src/services/compact/autoCompact.ts`。

有效窗口不是原始 context window，而是要先预留 summary 输出空间：

```typescript
export function getEffectiveContextWindowSize(model: string): number {
  const reservedTokensForSummary = Math.min(
    getMaxOutputTokensForModel(model),
    MAX_OUTPUT_TOKENS_FOR_SUMMARY,
  )
  // ...
  return contextWindow - reservedTokensForSummary
}
```

其中 `MAX_OUTPUT_TOKENS_FOR_SUMMARY` 是 20,000。  
源码注释里提到这个值是按高分位实际 compact summary 长度校出来的，不是随便取个整数。

再往下还有几组 buffer：

```typescript
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

这意味着对一个 200K 上下文模型，自动压缩不会等到 99% 才触发，而是会更早介入，给 summary 生成和回填留出空间。

## 5. 自动压缩最怕的是递归自爆

`shouldAutoCompact()` 里有很多 guard，最重要的是防 compaction 自己再触发 compaction。

例如：

- 当前 querySource 是 `compact` 就不要再 compact
- session memory 路径中不要再递归压缩
- 某些 reactive compact 实验路径要避免互相叠加

如果没有这些 guard，fork 出去做 compact 的子代理完全可能再次命中阈值，再 fork 一个 compact，再继续 fork，最后把自己打成风暴。

源码甚至有一个很扎眼的注释：  
过去有不少会话出现几十次、几百次、甚至三千多次连续 compaction failure，白白浪费大量 API 调用。于是他们加了 circuit breaker，连续失败 3 次后就停。

这说明"自动压缩"这事上线后远比想象中难。不仅要会压，还得会在压不动时及时认输。

## 6. Claude Code 其实有两套压缩路径

### Session Memory Compact

这是较新的路径。不是把整段对话重新总结，而是利用后台持续抽取出来的 session memory，只保留近期消息，再用 session memory 替代旧段。

优点：

- 更省 token
- 更快
- 不必每次都重新把整段历史喂给模型总结

### Full Conversation Compact

老路径，也更重。它会 fork 一个总结代理，把整段对话喂进去，要求产出结构化摘要。

这条路径的 prompt 很严格，要求 9 个部分，甚至先写 `<analysis>` 再写 `<summary>`。  
其中 `<analysis>` 会在最终结果里被剥掉，只保留摘要主体。

这套方法很实用：  
先逼模型按时间顺序做一遍深思考，可以提高摘要质量；再把 scratchpad 去掉，避免浪费上下文。

## 7. compact prompt 的设计非常"防工具误用"

总结 prompt 里有一段反复强调：

- 只能输出文本
- 不准调用任何工具
- 工具调用会被拒绝，而且只有一轮机会

为什么要写这么重？因为较新的模型在 adaptive thinking 下，哪怕明确说"不要调工具"，有时还是会手痒。  
而 compact 任务通常 `maxTurns: 1`，一旦它把唯一一轮浪费在工具调用上，整个压缩就直接失败。

这类防御很有意思，说明 Anthropic 已经在 prompt 里专门修补"模型在 meta-task 中也想继续用工具"的习惯。

## 8. 甚至支持部分压缩

Claude Code 不是只能"把前面都压掉"。它还支持 partial compaction：

- `from`：压缩某个点之后的消息，保留前缀
- `up_to`：压缩某个点之前的消息，保留后缀

这里的价值在于缓存。  
保留前缀时，prompt cache 更容易沿用；如果摘要插在前面，就更容易打掉下游缓存。

这再次说明，Claude Code 对 compaction 的目标不只是"把 token 降下来"，而是"降下来同时尽量别毁掉后续性能"。
