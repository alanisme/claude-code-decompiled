# Claude Code 的系统提示词工程：它不是一段提示词，而是一条装配线

很多人提到系统提示词，脑子里还是"一大段固定文本"。Claude Code 不是。它的系统提示词更像构建产物：静态规则、动态环境、工具说明、CLAUDE.md、记忆系统、MCP 说明、缓存边界，最后被组装成一个 `string[]` 数组，再送去 API。

从源码角度看，这已经不是 prompt writing，而是 **prompt engineering as software architecture**。

## 1. `getSystemPrompt()` 返回的不是字符串，而是分段数组

入口在 `src/constants/prompts.ts`。最终返回的结构大概是：

```typescript
return [
  getSimpleIntroSection(outputStyleConfig),
  getSimpleSystemSection(),
  outputStyleConfig === null || outputStyleConfig.keepCodingInstructions === true
    ? getSimpleDoingTasksSection()
    : null,
  getActionsSection(),
  getUsingYourToolsSection(enabledTools),
  getSimpleToneAndStyleSection(),
  getOutputEfficiencySection(),
  ...(shouldUseGlobalCacheScope() ? [SYSTEM_PROMPT_DYNAMIC_BOUNDARY] : []),
  ...resolvedDynamicSections,
].filter(s => s !== null)
```

这段代码里最重要的东西，不是具体哪一段，而是中间那个边界标记：

`__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__`

它把系统提示词硬切成两部分：

- 边界前：静态、跨用户可共享、适合 `cacheScope: 'global'`
- 边界后：动态、会话相关、通常只能 org 级或不缓存

这个设计非常聪明。因为系统提示词往往几千到上万 token，真正贵的不是"写出来"，而是每次请求都重新 prefill。Claude Code 显然在有意识地把高稳定度内容做成字节级可复用块。

## 2. 简化模式也有一条完全不同的快路径

如果设置了 `CLAUDE_CODE_SIMPLE`，系统会直接走一个极简提示：

```typescript
if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
  return [`You are Claude Code, Anthropic's official CLI for Claude.\n\nCWD: ${getCwd()}\nDate: ${getSessionStartDate()}`]
}
```

这说明维护者非常清楚：完整提示词很强，但也很重。  
所以他们保留了一个几乎"裸机"的模式，适合测试、最小开销场景，或者某些只想拿基本代理能力的环境。

## 3. 静态段落本质上是在定义 Claude Code 的人格和边界

边界前的几段，职责很清晰：

### `getSimpleIntroSection`

定义身份和基础安全策略，比如双用途安全约束、不能乱猜 URL 等。

### `getSimpleSystemSection`

解释系统机制：

- 模型输出会直接展示给用户
- 工具执行受 permission mode 控制
- 工具结果可能包含 prompt injection
- `<system-reminder>` 这类标签要按系统级信息理解

这其实很重要。很多代理系统默认假设模型"懂运行环境"，Claude Code 则反过来，把系统机制显式写进 prompt。

### `getSimpleDoingTasksSection`

这是行为规范最密的一段，也是 Claude Code 个性最强的地方。这有许多非常工程化的要求：

- 不要超需求发挥
- 不要为了"看起来更完善"加多余 fallback
- 不要为一次性操作造抽象

内部版还会插入更细的行为纠偏，比如少写注释、结束前先验证真的能跑。这些注释旁边经常带着 `@[MODEL LAUNCH]`，说明某些 prompt patch 是针对特定模型代的行为缺陷打的补丁。

换句话说，系统提示词已经成了**模型行为热修复层**。

## 4. 动态段落是一个注册式装配系统

边界后走的是 section registry：

```typescript
const dynamicSections = [
  systemPromptSection('session_guidance', () =>
    getSessionSpecificGuidanceSection(enabledTools, skillToolCommands)),
  systemPromptSection('memory', () => loadMemoryPrompt()),
  systemPromptSection('ant_model_override', () => getAntModelOverrideSection()),
  systemPromptSection('env_info_simple', () =>
    computeSimpleEnvInfo(model, additionalWorkingDirectories)),
  systemPromptSection('language', () => getLanguageSection(settings.language)),
  systemPromptSection('output_style', () => getOutputStyleSection(outputStyleConfig)),
  DANGEROUS_uncachedSystemPromptSection('mcp_instructions', () =>
    isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(mcpClients),
    'MCP servers connect/disconnect between turns'),
  systemPromptSection('scratchpad', () => getScratchpadInstructions()),
  systemPromptSection('frc', () => getFunctionResultClearingSection(model)),
  systemPromptSection('summarize_tool_results', () => SUMMARIZE_TOOL_RESULTS_SECTION),
]
```

这套机制最妙的地方在于：**section 是一等公民**。每段都有名字、计算函数和缓存语义。

普通 section 会被 memoize，直到 `/clear` 或 `/compact` 才失效。  
只有 `DANGEROUS_uncachedSystemPromptSection` 会每轮重算。

这个命名很直接，甚至带点"敬畏"的意味。因为 prompt cache 被破坏的代价很高，必须把"我真的要每轮重算"视作高风险行为。

## 5. 环境信息不是装饰，而是在减少模型瞎猜

`computeSimpleEnvInfo()` 会构造一个环境块，大致包含：

- 主工作目录
- 是否 git 仓库
- 平台 / shell / OS 版本
- 当前模型名和模型 ID
- 知识截止时间
- 最新模型家族说明

别小看这段。它直接影响模型：

- 怎么写路径
- 是否预设 git 工作流
- 是否考虑 shell 差异
- 是否误以为自己知道某个未来版本

如果当前目录是 git worktree，提示词里还会明确说：**不要 `cd` 回原仓库根目录**。  
这已经不是普通提示，而是对常见代理失误的定点修补。

## 6. 工具说明本身也是提示词系统的一部分

每个工具目录下通常都有 `prompt.ts`。这些提示不是一句"这个工具能干嘛"，而是相当具体的使用约束。

例如 FileEditTool 会明确告诉模型：

- 先 Read 再 Edit
- 保留精确缩进
- 尽量修改现有文件，不要随便新建
- `old_string` 不唯一会失败

内部版还会额外说：`old_string` 选 2-4 行唯一上下文就够，不要夹十几行冗余内容。

这说明 Anthropic 不是只在系统提示词顶层教模型怎么干活，而是把"工具操作细则"拆到各工具自身，形成局部专用指导。

从可维护性上说，这比把所有工具说明塞进一个超级大 prompt 文件靠谱得多。

## 7. CLAUDE.md 是一套分层注入系统

`src/utils/claudemd.ts` 会按优先级加载：

1. `/etc/claude-code/CLAUDE.md`
2. `~/.claude/CLAUDE.md`
3. 项目里的 `CLAUDE.md` / `.claude/CLAUDE.md` / `.claude/rules/*.md`
4. `CLAUDE.local.md`

而且支持：

- `@include`
- YAML frontmatter 的 `paths`
- HTML comment stripping
- exclude patterns

这已经远超"读一个本地规则文件"的程度了。它更像一个层级化的 instruction overlay system。

源码里注入 CLAUDE.md 内容时用的措辞也非常强硬：

> 这些指令覆盖默认行为，你必须严格遵守。

这等于在系统提示词里再加一层"本地最高优先级策略"。

## 8. Memory 不是一个文件，而是两套机制

Claude Code 里至少有两类 memory：

### 用户/项目显式记忆

也就是前面说的 CLAUDE.md 体系，由人维护。

### 自动记忆系统

在 `src/memdir/` 下，Claude Code 会维护一个 `MEMORY.md` 索引，加上分主题文件。`MEMORY.md` 本身会进入系统提示词，但被严格限制：

- 最多 200 行
- 最多 25KB

这个限制非常关键。否则模型或用户很容易把 index 撑成一坨没法用的大杂烩，把上下文窗口白白吃掉。

从源码可以看出，Anthropic 已经意识到：memory 系统最大的敌人不是"记不住"，而是"越记越肥，最后拖死 prompt"。

## 9. 内部用户还有一层"远程热补丁"

对内部用户，`defaultSystemPromptSuffix` 可以在系统提示词后面再追加内容，相当于不发版也能打 prompt patch。

这件事影响很大，因为它意味着：

- 某些行为调整不需要重新部署客户端
- 提示词本身已经被当作可运维资产
- 线上观察到问题后可以快速缓解

说白了，Claude Code 的 prompt 已经不只是"模型说明书"，而是产品控制面的一部分。

## 10. `Undercover` 模式连提示词也一起"去 AI 味"

如果 `isUndercover()` 为真，系统会从提示词里剥掉模型名、内部代号等敏感识别信息。

这一步很说明问题。因为 Anthropic 显然意识到：  
如果模型把自己的代号、版本或"我是 AI 编码助手"那套身份标签顺着 commit message、PR 描述带出去，就会直接破坏 undercover 的目标。

所以他们不仅在输出层隐藏 AI 痕迹，连系统提示词输入层也一起处理。

## 11. 对这套 prompt 工程的判断

Claude Code 的系统提示词体系最厉害的，不是文案写得多好，而是它有明确的软件工程特征：

- 分段装配
- 明确缓存边界
- 动态 section 注册
- 细粒度失效策略
- 工具提示模块化
- 本地规则分层覆盖
- 远程 patch 能力

如果想在自己的 agent 项目中吸取经验，最值得学的是这几点：

1. 不要把 prompt 当纯文本，要当构建产物
2. 静态和动态内容必须分层，不然缓存命中率会被打爆
3. 工具说明应该局部模块化，而不是大锅炖
4. memory 一定要有硬上限
5. 模型行为补丁要允许热修，不要全部绑死在代码版本里

Claude Code 的系统提示词，本质上已经不是"写 prompt"，而是在做一条**面向缓存、面向会话、面向运维的提示词供应链**。
